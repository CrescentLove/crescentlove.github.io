KVM&QEMU虚拟化

发表于 2022-12-24 分类于 kvm
KVM&QEMU虚拟化

KVM内核实现CPU&内存的虚拟化，QEMU实现IO虚拟化，通过IOCTL与KVM内核交互。

一个典型的KVM架构包含以下：
Guest：客户机系统，包括CPU（vCPU）、内存、驱动（Console、网卡、I/O 设备驱动等），被 KVM 置于一种受限制的 CPU 模式下运行。
KVM：运行在内核空间，提供CPU 和内存的虚级化，以及客户机的 I/O 拦截。Guest 的 I/O 被 KVM 拦截后，交给 QEMU 处理。
QEMU：修改过的为 KVM 虚机使用的 QEMU 代码，运行在用户空间，提供硬件 I/O 虚拟化，通过 IOCTL /dev/kvm 设备和 KVM 交互。

KVM

KVM模块加载的时候，首先初始化内部的数据结构；然后KVM模块检测当前的CPU，打开 CPU控制及存取CR4的虚拟化模式开关，并通过执行VMXON指令将VMM置于VMX root；最后，KVM 模块创建特殊设备文件 /dev/kvm 并等待来自用户空间的指令。‘
这里提到的VMX root和VMX non-root模式均由Intel-VT硬件辅助虚拟化技术提供。VMM运行在根模式，guest运行在非根模式。

CPU虚拟化

在KVM中每个Guest均是一个Qemu进程，而每个Guest都会被分配若干个vCPU，其中每个vCPU都是该QEMU进程下派生的一个线程。那么vCPU怎么和pCPU对应呢？

pCPU（socket）对应CPU，每个CPU有多个core，开启Hyper-Threading后每个core对应两个线程。一般来说pCPU下开的线程一一对应vCPU。（了解一下vCPU超分）

vCPU超分Over-Commit

在实际场景里，服务器通常会提供QoS服务，只要能满足QoS，我们完全可以为Guest提供远超物理层面的资源，比如此处的vCPU超分：给Guest提供的vCPU总数超过pCPU的总数。

常用做法：对多个单CPU的Guest提供Over-Commit。这是因为在负载不高的情况下，单CPU的Guest能够被VMM高效的调度。

不推荐做法：给单个Guest提供超过pCPU总数的vCPU数量，相当于每个vCPU都在挨饿。

Processor Affinity

首先通常情况下，进程和线程的处理器亲和性对每个pCPU都是一致的，在linux 默认的CFS进程调度算法下，如果QEMU进程在不同pCPU间切换的话，就会有一个缓存命中失效的问题。通过把进程绑定到特定的CPU上可以有效减少这样的资源消耗。
当然绑定pCPU也是有弊端的，本来就不多的pCPU还被强占了几个，如果这几个Guest负载还不高的话，那浪费就太严重了。所以在使用亲和性的时候最好还是综合考虑负载均衡和缓存命中失效的资源消耗。

内存虚拟化

虚拟化技术引入后，内存地址空间更加复杂了，客户机(Guest)和宿主机(Host)都有自己的地址空间:

  • GVA: Guest虚拟地址
  • GPA: Guest物理地址
  • HVA: Host虚拟地址
  • HPA: Host物理地址

显而易见，Guest负责GVA和GPA之间的转换（读写CR3）；Host负责HVA和HPA之间的转换；而GPA和HPA之间的转换，就需要虚拟化层(Hypervisor)辅助了，这个过程一般被称为内存虚拟化。

为了解决GVA-GPA-HPA的转换关系，在没有硬件辅助的时代，Hypervisor通过影子页表，很巧妙的将GVA-GPA映射到GVA-HPA, 功能虽然达成，但是在很多实际场景下，如进程频繁切换，内存频繁分配释放等，性能损耗会非常大；EPT在硬件的帮助下，实现内存虚拟化简单直接，传统页表继续负责GVA-GPA, 而EPT负责GPA-HPA; 虽然内存访问延时可能会增加一些，但是大幅减少了因为页表更新带来的VMExit, 综合性价比提升巨大, 所以现代内存虚拟化，基本都被EPT统一了。

  • 页表更新不需要VMExit了？是的，引入EPT后，Guest读写CR3寄存器或Guest Page Fault，执行INVLPG指令等不会触发VMExit。

    如果 GPA 地址为空，即缺页，则 CPU 产生缺页异常，注意，这里，如果是软件实现的方式，则会产生 VM-exit，但是EPT硬件实现方式，并不会发生 VM-exit，而是按照一般的缺页中断处理，这种情况下，也就是交给 Guest 内核的中断处理程序处理。

    在中断处理程序中会产生 EXIT_REASON_EPT_VIOLATION，Guest 退出，VMM 截获到该异常后，分配物理地址并建立 GVA 到 HPA 的映射，并保存到 EPT 中，这样在下次访问的时候就可以完成从 GPA 到 HPA 的转换了。

  • 我们通常会用TLB缓存GPA-HPA的映射，但是切换vCPU时（相当于是切换线程，刷新缓存）TLB会被刷新，解决办法是Intel的VPID技术。把每个TLB唯一对应一个vCPU，有效减少vCPU切换对TLB的影响，【上下文切换消耗变少了】。

影子页表

在影子页表和EPT出现以前，即使是每次访问内存地址都需要VMM参与，软件进行多次的转换地址操作。影子页表直接做了个GVA-->HPA的映射。

要实现这样的映射，必须为 Guest 的系统页表设计一套对应的影子页表，然后将影子页表装入 Host 的 MMU 中，这样当 Guest 访问 Host 内存时，就可以根据 MMU 中的影子页表映射关系，完成 GVA 到 HPA 的直接映射。而维护这套影子页表的工作则由 VMM 来完成。

由于 Guest 中的每个进程都有自己的虚拟地址空间，这就意味着 VMM 要为 Guest 中的每个进程页表都维护一套 对应的影子页表，当 Guest 进程访问内存时，才将该进程的影子页表装入 Host 的 MMU 中，完成地址转换。

我们也看到，这种方式虽然减少了地址转换的次数，但本质上还是纯软件实现的，效率还是不高，而且 VMM 承担 了太多影子页表的维护工作，设计不好。

内存超分Over-Commit

类比“虚拟内存分配远大于物理内存”&“磁盘上开辟缓存区”

  • 内存交换（Swapping）：交换到swap分区
  • 气球（Ballooning）：virtio_balloon动态调整host和guest的内存空间
  • 页共享（Page Sharing）：因为多个guest相当于是多个qemu进程，会有很多相同内存页。KSM功能扫描相同内存页进行合并，附加标志位Copy On Write。也就是在要脏页的时候赶紧复制一份去写操作。

KSM

展开说一下上面的页共享（Page Sharing），KSM是为KVM量身定做的内存共享技术，因为KVM里每个guest都是一个QEMU进程，且多数情况下运行相同的操作系统和程序，天然的对于每个进程而言，会有很多相同的内存页，通过KSM页共享，使得缓存命中率有效提高。

实现方法：KSM使用一个或多个进程持续扫描运行中的应用进程，并共享他们相同的内存页，附加标志位COW（Copy On Write）。

这又会有几个问题：

  • 使用KSM就需要起进程扫描，一定会占用CPU宝贵的处理时间，相当于是用CPU使用率换内存使用率。
  • 使用KSM还得能分辨出哪些内存是不经常更新的，否则会使得“COW复制新内存页供修改”的操作消耗大量内存和CPU资源。
  • 使用KSM配合内存超分需要注意swap分区大小，遇到高COW负载的时候可能物理内存会不够用。

一个典型的KVM调用流程

  • guest执行特权指令
    graph LR
guest[guest]--"触发VMExit"-->call[执行VMCALL]
call-->exec[VMM处理]
exec-->resume[执行VMRESUME]
resume--"触发VMEntry"-->guest

在VMExit和VMEntry的上下文切换时，我们也需要类似PCB/TCB这样的控制块来保存上下文切换所需的信息，在KVM里VMX提供了VMCS结构。其中control data保存触发模式切换的事件及原因；Guest state 保存Guest运行时状态，在VMEntry时加载；Host state保存VMM运行时状态，在VMExit时加载。

QEMU

根据之前VM执行特权指令流程，如果guest频繁发起IO操作请求，系统会一直触发VMExit/VMEntry，一直在VMX root和non-root模式下切换，资源消耗大。

我们可以通过virtio驱动实现IO设备半虚拟化。VM里部署前端驱动，QEMU里部署后端驱动，在这两层之间实现一个环形缓冲区virtio-ring，保存IO请求与执行信息。

  • 网络传输优化：网络协议栈运行在内核态，那么virtio在QEMU里（用户进程）部署后端驱动处理网络请求仍然需要从用户态切换到内核态。通过在内核里加入vhost-net驱动，可以直接接管VM传递出来的网络IO操作，避免了内核态（网络栈）和用户态（QEMU）的切换。
  • 云计算优化：大规模使用的OVS和CDN仍然会带来大量的内核态（网络栈）与用户态（CDN/OVS进程）切换。通过使用vhost-user，vSwitch与DPDK可以在用户态完成网络数据包交换处理，从而大幅提升了网络虚拟化性能。

除了上面的技术，Intel还提供了下面几种方案：

  • Intel-VT硬件辅助虚拟化
  • DPDK：结合vhost-user&vSwitch，可以在用户态完成网络数据包处理
  • SPDK：结合UIO和PMD，通过一个VRing绕开KVM和QEMU实现用户态进行存储管理。（UIO实现了除少部分共性驱动在内核态，其他大多数设备IO在用户态）

更多值得一看的技术

SPDK

spdk的基石是用户态（user space）、轮询（polled-mode）、异步（asynchronous）、无锁（lockless）的NVMe驱动

  1. 将存储用到的驱动转移到用户态，从而避免系统调用带来的性能损耗，顺便可以直接使用用户态内存落盘实现零拷贝

  2. 使用polling模式

    1. 轮询硬件队列，而不像之前那样使用中断模式，中断模式带来了不稳定的性能和延时的提升
    2. 任何业务都可以在spdk的线程中将轮询函数注册为poller，注册之后该函数会在spdk中周期性的执行，避免了epoll等事件通知机制造成的overhead。
    3. 独占一个core，不允许有进程上下文切换

  4. 避免在IO链路上使用锁。使用无锁队列传递消息/IO

    1. spdk设计的主要目标之一就随着使用硬件（e.g. SSD，NIC，CPU）的增多而获得性能的线性提升，为了达到这目的，spdk的设计者就必须消除使用更多的系统资源带来的overhead，如：更多的线程、进程间通信，访问更多的存储硬件、网卡带来的性能损耗。
    2. 为了降低这种性能开销，spdk引入了无锁队列，使用lock-free编程，从而避免锁带来的性能损耗。
    3. spdk的无锁队列主要依赖的dpdk的实现，其本质是使用cas（compare and swap）实现了多生产者多消费者FIFO队列。有关无锁队列的实现可以看这篇文章

进程上下文切换原因：

  • cpu时间片耗尽
  • 进程在系统资源不足（比如内存不足）时，要等到资源满足后才可以运行，这个时候进程也会被挂起，并由系统调度其他进程运行。
  • 进程主动调用sleep等函数让出cpu使用权。
  • 当有优先级更高的进程运行时，为了保证高优先级进程的运行，当前进程会被挂起，由高优先级进程来运行。
  • 硬件中断会导致CPU上的进程被挂起，转而执行内核的中断服务程序。

透明大页THP

大页

通过在内核编译的时候使能CONFIG_HUGETLBFS和CONFIG_HUGETLB_PAGE，能够支持2M、1G的大页。

优点：

  • 使用大页使得页表数量变少，占用的内存也变少了
  • 页表够大，cover住的内存也比较多，地址转换少了（储存地址转换信息的CPU缓存消耗变少），所以TLB缓存失效也变少了。

缺点：

  • 大页必须在使用前准备好，一般是linux启动时初始化大页池（后期因为内存碎片多的原因可能无法成功分配大页）（如果没用到不就白白占用物理内存了么，很僵硬）
  • 大页必须常驻物理内存，不允许swap，也不允许ballooning
  • 需要使用root权限挂载hugetlbfs大页系统，应用程序必须显式使用大页（mmap、shmget调用）

透明大页

针对大页的缺点，首先THP对所有应用透明（躺平由操作系统为内存管理费劲），其次THP是可交换的（swap的时候会分散成4KB小页），最后，内核线程khugepaged会在内存充裕的时候不断扫描，把常规物理页装换到透明大页中（仅匿名内存）。

补充知识

特权等级

x86架构的操作系统直接运行在硬件上，分为内核态和用户态。指定了4种特权级别的运行状态Ring0 - Ring3，其中Ring 0是内核态，运行OS内核，权限最高，可执行特权指令（对系统资源进行管理与分配等）或执行敏感指令（如读写时钟、控制中断、修改内存页表、访问地址重定位系统，以及所有IO指令等）。Ring3是用户态，运行用户应用程序，只能执行非特权指令；需要执行特权或敏感的特权指令时通过系统调用方式从Ring3用户态切换到Ring0内核态，内核完成相关操作后再切换返回。Ring 1、Ring2保留使用。
在KVM里，当guest执行特权指令时，触发VMExit到VMM，处理完后再触发VMEntry回到guest。
